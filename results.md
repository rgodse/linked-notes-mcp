# Retrieval Benchmark — Measured Results

A local, reproducible benchmark for **how well this server retrieves the right notes for a query** — because the memory field's public benchmarks are contested and non-reproducible, and "it feels better" is not a number.

## What was measured

- **Task:** given a natural-language query, retrieve the note(s) that answer it.
- **Metric:** **recall@5** (did the gold note(s) land in the top 5), primary; **MRR**, secondary.
- **Confidence:** bootstrap 95% CIs (2000 resamples, seeded) on every number, so "is this difference real?" is answerable, not eyeballed.
- **Dataset:** a **60-note synthetic software-domain vault + 40 query→gold-note pairs**, generated by `openrouter/anthropic/claude-haiku-4.5` and **committed** under [`evals/dataset/`](evals/dataset) so these numbers reproduce with no spend. Gold labels are the notes the generator was told the answer lives in.
- **Rungs:** `text` (lexical search + graph expansion) → `+ salience` (rerank by note importance) → `+ hybrid` (fuse text with FastEmbed vector search).

## The headline: the benchmark caught a real bug

The first run reported `text: recall@5 = 0.00`. That is not a weak baseline — it is a **broken** one, and the benchmark is what surfaced it:

`KnowledgeGraph.search` matched the **entire query string as a substring** against each note field. A single keyword (`"JWT"`) worked; a natural-language question (`"What authentication method is used for stateless auth?"`) matched nothing and returned an empty list. **The shipped `get_context` tool returned nothing for exactly the kind of query an agent actually asks.**

Fixing search to **tokenize** the query (score per term, keep a whole-phrase bonus) took recall@5 from **0.00 → 0.81**. That tokenization fix — not any fancy retrieval — was the single biggest retrieval improvement in this work, and it shipped in the server.

## Results (after the fix)

| Rung | recall@5 | 95% CI | MRR |
|---|---|---|---|
| text (tokenized) | **0.81** | 0.71–0.90 | 0.82 |
| + salience | 0.81 | 0.71–0.90 | 0.82 |
| + hybrid (embeddings) | 0.82 | 0.72–0.91 | 0.84 |

## The honest finding: salience and embeddings showed no measurable lift here

All three rungs sit inside each other's confidence intervals. Salience reranking is **measurably neutral** on this vault; hybrid embeddings add **+0.01** (noise). On this benchmark, **well-tokenized lexical search is already good enough**, and the extra machinery does not earn its place.

**Why — and this is a limitation, not a triumph:** the 40 queries were written by the *same model* that wrote the 60 notes, so they **share vocabulary**. Lexical search saturates when the query uses the note's own words, which leaves embeddings nothing to recover. On a real vault where users ask in **different words than the notes use** (the paraphrase gap embeddings exist to close), hybrid would be expected to help — but *this* benchmark, by construction, cannot demonstrate that. Reporting the null honestly is the point; manufacturing a win would defeat the purpose of having a benchmark at all.

## What shipped as a result

- **Tokenized search** in `KnowledgeGraph.search` — the real fix (`get_context` now answers natural-language queries). This is a product change, not a study artifact.
- **Salience reranking** in `get_context` (relevance + note `importance`) — cheap, neutral on this benchmark, kept as a sensible default that can help on importance-weighted vaults. Zero new dependencies.
- **Optional hybrid retrieval** via a rebuildable FastEmbed index (`pip install 'linked-notes-mcp[embeddings]'`) — local ONNX model, no API/LLM, files remain the source of truth. Available for vaults where the paraphrase gap is real; the server falls back to text-only when the extra isn't installed.

## Limitations

1. **Single synthetic domain** (a software system). Results may not transfer to arbitrary real vaults.
2. **Same-generator queries share vocabulary with notes**, which understates embeddings (see above). A paraphrase-adversarial query set is the honest next step to give hybrid a fair test.
3. **Gold labels are as good as the generator.** A query whose answer also lives in an unlabeled note would understate recall.
4. **A DSPy-optimized rung is deferred.** The benchmark is built to accept one (query→retrieval-params, scored by the same recall@5) without changing the harness.

## Reproduce

```bash
uv sync --extra embeddings           # FastEmbed (local, no API)
# dataset is committed; to regenerate it you also need litellm + a key:
#   uv pip install litellm && OPENROUTER_KEY=... uv run python -m evals.gen --force
uv run python -m optimize.run         # prints the rung table above
```

Everything reads from `evals/dataset/` and the seeded CIs, so the table reproduces exactly. Model id and counts are in [`evals/dataset/meta.json`](evals/dataset/meta.json).
